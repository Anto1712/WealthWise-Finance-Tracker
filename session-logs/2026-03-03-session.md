## Where the Agent Got Stuck

- **Background process management in bash**: Attempted to start the API in the background with `&`, check it with `sleep + cat log`, kill ports with netstat/awk. These commands either timed out, couldn't find `tail`, or left zombie processes holding port 4000. The shell environment (git bash on Windows with limited coreutils) is hostile to this pattern.
  - **How resolved**: User interrupted; gave explicit terminal commands instead.
  - **Suggestion**: Never attempt to manage long-running background processes in the bash tool on Windows. Either use `run_in_background: true` on the Bash tool (which uses a proper async mechanism) or just give the user the commands up front.

- **`find` on Program Files rejected**: First attempt to locate `node.exe` used `find /c/Program\ Files` which was rejected by the user's tool permission settings.
  - **How resolved**: Used a targeted `ls` path instead.
  - **Suggestion**: On Windows, prefer `ls "/c/Program Files/nodejs/"` and similar targeted paths over recursive `find` which triggers permission prompts.

- **Port 4000 already in use**: Multiple background API starts during debugging left processes holding port 4000. The cleanup attempts via netstat/taskkill were unreliable in git bash.
  - **How resolved**: User ran `taskkill` manually; then got clean boot.
  - **Suggestion**: Add port-availability check to `dev.ts` startup (try binding a test socket before starting) and print a clear "kill existing process" message if port is taken.

- **`cmd.exe /c "where node"` rejected**: Used Windows-specific `cmd.exe` call to find npm, which was rejected.
  - **How resolved**: Used bash-compatible path checks instead.
  - **Suggestion**: On Windows with git bash, use `/c/...` paths and `ls` for discovery rather than `cmd.exe` invocations.
